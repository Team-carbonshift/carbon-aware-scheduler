# 탄소 인지 로드밸런서 — 전체 정리 (백업용)

> LSTM 탄소강도 예측 + ILP 최적화로 요청을 지역별로 라우팅하는 시스템.
> 목표: **레이턴시 SLO를 지키면서, 지역 용량 안에서, 총 탄소 배출을 최소화**하는 라우팅 규칙을 매 주기 산출.

## 1. 전체 파이프라인 (한눈에)

```
관측(M, λ) → LSTM 탄소 예측(M̂) → 정규화(M̃, l̃)
   → ILP로 최적 배분 개수 x* → 비율 r로 변환 → 다음 구간 적용 → 반복
```

| 단계 | 하는 일 | 산출물 |
|------|---------|--------|
| 1 | 관측 (탄소강도, 요청량) | M_(j,t), λ_(i,t) |
| 2 | LSTM 예측 | M̂_(j,t+1) (8개) |
| 3 | 정규화 (0~1) | M̃ (8개), l̃ (64개) |
| 4 | ILP 풀기 (PuLP/CBC) | x* (64개 정수) |
| 5 | 비율 변환 | r_ij (64개, 행합=1) |
| 6 | 적용 | 다음 구간 라우팅 |

---

## 2. 모든 공식

### 1단계 — 관측 (매 주기 t)
```
M_(j,t)      : j지역 현재 탄소강도 (탄소 API)
λ_(i,t)      : 출발지 i의 직전 1시간(t-1~t) 누적 요청 수 (LB 로그)
```
둘 다 모자(^) 없음 = 관측치.

### 2단계 — LSTM 예측
```
M̂_(j,t+1) = LSTM( M_(j,t-23), …, M_(j,t) )
```
과거 24시간 탄소강도 → 다음 시간 예측. 지역 8개 → 예측값 8개. 모자(^) = 예측치.

### 3단계 — 정규화 (0~1)
```
M̃_(j,t+1) = M̂_(j,t+1) / max_k M̂_(k,t+1)      ← 탄소, 8개, 매주기 갱신
l̃_ij      = l_ij / max_(i,j) l_ij  = l_ij / 244  ← 레이턴시, 64개, 고정
```
단위가 달라서(g/kWh vs ms) 각각 최댓값으로 나눠 0~1로 맞춤 → α가 공정하게 작동.

### 4단계 — ILP

**목적함수:**
```
min_x  Σ_i Σ_j ( α·M̃_(j,t+1) + (1-α)·l̃_ij ) · x_(ij,t+1)
```

**제약조건:**
```
① 지역 용량 :  Σ_i x_(ij,t+1) ≤ C_j = c·(s_j^act + s_j^idle)   ∀j
② SLO 안전망:  x_(ij,t+1) · (l_ij - L_max) ≤ 0                  ∀i,j
③ 전량 처리 :  Σ_j x_(ij,t+1) = λ_(i,t)                          ∀i
④ 정수·비음수:  x_(ij,t+1) ∈ ℤ_≥0
```
→ PuLP로 적고 CBC로 풀면 최적해 **x\*** (64개 정수).

### 5단계 — 라우팅 비율 변환
```
r_ij = x*_(ij,t+1) / λ_(i,t)
```
행 합 Σ_j r_ij = 1 (제약③ 덕분에 자동). 요청량이 t+1에 달라져도 비율이라 흡수.

### 6단계 — 적용
```
다음 구간 동안 r_ij 비율로 분배 (Traefik 등) → 다음 주기에 1단계로 복귀
```

## 3. 기호 정의 (미지수는 x 하나, 나머지 다 입력)

| 기호 | 종류 | 의미 |
|------|------|------|
| **x_(ij,t+1)** | 미지수 | i 요청 중 j가 처리할 **개수** (정수, 64개). ILP 결과. |
| **r_ij** | 파생 | x\*/λ, **비율** (0~1). 최종 산출물, LB에 꽂는 값. |
| **M̂_(j,t+1)** | LSTM 출력 | j지역 다음 시간 예측 탄소강도 (gCO₂/kWh). 매주기 변함. |
| **l_ij** | 측정 상수 | i→j 레이턴시 (ms). max=244. **시간 불변.** |
| **λ_(i,t)** | 관측 | i의 직전 1시간 누적 요청량. |
| **C_j** | 정한 상수 | j 처리 한도 = c·(s_act+s_idle). (안전: ×0.8 headroom) |
| **α ∈ [0,1]** | 정책 다이얼 | 탄소↔레이턴시 교환비. 실험에서 스윕 {0,.25,.5,.75,1}. |
| **L_max** | 정책 다이얼 | 레이턴시 SLO 상한(ms). 넘는 경로는 ②에서 x=0 강제. |

## 4. 핵심 개념 (헷갈리기 쉬운 것)

### x vs r
- **x = 개수** (ILP 계산용, 정수). 예: x_(B→A)=500 → "B 요청 500개를 A가 처리".
- **r = 비율** (실제 적용용, 0~1). 예: r_(B→A)=0.5 → "B 요청의 50%를 A로".
- **왜 변환?** x는 현재 λ 기준 개수라, t+1에 요청량이 바뀌면 어긋남.
  비율로 두면 요청량이 출렁여도 배분 의도(50%)가 유지됨.

### 시간 첨자 t+1 규칙
- **예측/미래에서 온 값 → t+1 붙음**: x_(ij,t+1), M̂_(j,t+1), M̃_(j,t+1)
- **고정 상수 → 첨자 없음**: l_ij, l̃_ij (레이턴시는 시간 불변)
- 결론: **시간을 타는 재료는 탄소(M̃)뿐.** 레이턴시가 고정이라
  "탄소가 안 변하면 라우팅 표도 거의 안 변함".

### Σ_i Σ_j 의 의미 (표로 이해)
- 행 = 출발지 i, 열 = 목적지 j. 8×8 = 64칸.
- **Σ_i Σ_j (둘 다)** = 표 전체 = 총비용 (목적함수)
- **Σ_j (j만)** = 가로 한 줄 = 한 출발지가 내보낸 요청 합 (제약③)
- **Σ_i (i만)** = 세로 한 칸 = 한 목적지가 받은 요청 합 (제약①)

### x는 "방향"이 아니라 "개수"
- x_(ij)는 "i가 j로 간다(예/아니오)"가 아님.
- **몇 개 보내는지**. 한 출발지가 여러 목적지로 쪼개질 수 있음
  (독일 1000개 = 프랑스 500 + 독일 자기 500).

### SLO는 어디에? → 제약②의 L_max
- 목적함수의 l̃ = "빠르면 좋다" (부드러운 선호, 협상 가능)
- 제약②의 L_max = "이 선은 절대 못 넘는다" (딱딱한 상한, SLO)
- 같은 레이턴시인데 역할이 다름. 운전으로 치면 연비(선호) vs 제한속도(SLO).

### 결과물 = 매 주기 8×8 표 1장
- "8×8개의 표"가 아니라 **"8×8칸짜리 표 1장"**.
- 각 칸 = i→j 비율. 각 행 = 한 출발지의 분배 규칙.
- 탄소 변화 적으면 시간대별 표끼리 비슷할 수 있음.

## 5. ILP / PuLP / CBC 관계

| 이름 | 층위 | 역할 |
|------|------|------|
| **ILP** | 문제의 종류 | Integer Linear Programming. 목적·제약이 다 1차식(L), 변수 정수(I), programming=최적화(P). 정수 제약 때문에 NP-hard. |
| **PuLP** | 파이썬 라이브러리 | 수식을 파이썬으로 옮겨 적는 모델링 도구(통역사). 직접 풀지 않음. |
| **CBC** | 솔버 엔진 | 실제로 계산해 x\*를 찾음(계산기). Branch(소수해를 정수로 쪼갬)+Cut(탐색공간 가지치기). 무료, PuLP에 기본 포함. 64변수엔 순식간. |

흐름: **ILP**(문제 종류) → **PuLP**(코드로 표현) → **CBC**(실제 계산).

## 7. 염려사항 (우선순위순)

| # | 염려 | 왜 문제 | 심각도 | 대응 |
|---|------|---------|--------|------|
| 1 | **Infeasibility** | 제약①+③ 충돌(Σλ>ΣC) 시 해 없음 → 표 안 나옴 → LB 멈춤 | **높음** | slack 변수 or 직전 표 재사용 |
| 2 | **쏠림현상** | 선형이라 제일 싼 곳 몰빵 → 과부하·연쇄붕괴·flip-flop | 중~높음 | C_j headroom(0.8) |
| 3 | **적용시점 용량 초과** | r은 출발지 방향만 보장, t+1 트래픽↑시 목적지 과부하 | 중간 | 적용단 클램프 or headroom |
| 4 | **LSTM 예측 오차** | 예측 틀리면 엉뚱한 지역으로 | 중간 | 모니터링, 급변시 baseline 폴백 |
| 5 | **정규화 비정상성** | 매주기 max 바뀌어 α 의미 미세 변동 | 낮음 | max를 고정상수로 |
| 6 | **콜드스타트 무시** | s_idle "즉시 기동" 낙관 가정 | 낮음 | future work |

### 쏠림 상세 (연쇄 과정)
```
쏠림 → ①깨끗한 지역 집중(의도됨) → ②부하로 실제 레이턴시↑(모델은 상수로 오판)
→ ③용량 초과→요청 드롭 → ④넘친 트래픽 옆 지역으로→연쇄붕괴 → ⑤+예측 흔들림→flip-flop
```

## 8. 채택한 개선안

### (A) 성급한 일반화 → MPC (Receding Horizon)
**문제:** 한 번 예측으로 1시간 통째 확정 = open-loop, 성급함.
**해결:** 매 15분마다 "앞으로 1시간" 예측·최적화하되 **앞 15분만 적용**,
15분 뒤 새 관측으로 다시. + 예측이 실제와 크게 어긋나면 즉시 재최적화(이벤트 기반).
→ 멀리 보는 계획성은 유지, 오래 밀어붙이는 단점은 제거. 주기는 탄소 API 해상도에 맞춤(5~15분).

### (B) 쏠림 → headroom + slack (세트)
```
headroom : C_j = int( 0.8 · c · (s_act + s_idle) )   ← 용량 여유, 분산 유도
slack    : Σ_j x_ij + s_i = λ_i,  목적함수에 +1000·s_i  ← 못 받으면 페널티, 항상 feasible
```
- headroom이 쏠림 완화 + 적용시점 과부하 여유 (염려 2,3)
- slack이 headroom 유발 infeasibility 방어 (염려 1)
- **둘이 상충을 상쇄** → 반드시 세트로.
- **목적함수는 선형(ILP) 유지** → CBC로 계속 풀림. (2차 페널티는 MIQP 되어 솔버 교체 필요 → 비채택, future work)

**튜닝:** headroom 0.8(더 낮추면 분산↑ 탄소이득↓), slack 페널티는 단가(0~1)보다
훨씬 크게(1000) → "웬만하면 다 받되 정 안 되면 포기" 순서 보장.

### 레이턴시 고정 결정
- 부하 데이터 없음 + 공식 문서가 baseline 고정값 제공 + 핵심은 탄소 라우팅.
- **레이턴시는 상수로 유지가 정당.** 부하로 인한 과부하는 레이턴시가 아니라 **C_j로 방어.**
- 부하-레이턴시 동적 모델링은 future work (친구 스케줄러 실시간 부하 s_j 연동 시).

## 9. 발표용 한 줄 요약

> "LSTM으로 지역별 탄소강도를 예측하고, ILP로 레이턴시 SLO·용량 제약 하에 탄소 최소화 라우팅을 매 주기 산출한다. Open-loop의 성급함을 피하기 위해 receding horizon(MPC)을 채택하고, 쏠림은 용량 headroom으로 완화하며 그로 인한 infeasibility는 slack 변수로 방어한다. 레이턴시는 지리적 baseline으로 고정하고 부하 기반 동적 모델링은 향후 과제로 둔다."

## 10. 실험 아이디어

- **α 스윕** {0, 0.25, 0.5, 0.75, 1} → 탄소 vs 레이턴시 파레토 곡선
- **L_max 스윕** {100, 150, 200, 300} → SLO 엄격도 vs 라우팅 자유도
- **갱신 주기 스윕** {5, 15, 30, 60분} → 교정성 vs 안정성(flip-flop)
- **headroom 스윕** {0.7, 0.8, 0.9, 1.0} → 분산 vs 탄소이득
- **용량 축소 시나리오** → 쏠림이 2번째 깨끗한 지역으로 분산되는지 관찰
